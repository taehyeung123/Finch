# 인스타 댓글 자동 DM — 비용·리스크 점검 보고서

이 문서는 "인스타 댓글 자동 DM" 기능을 실제로 운영했을 때 발생하는 **서버·저장 비용, 기술 실패 모드, 정책·법률 리스크, 운영/UX 엣지케이스**를 한곳에 정리한다. Meta 2026 개발자 문서와 국내법(정보통신망법·개인정보보호법)을 근거로 하며, 각 항목마다 제품이 해야 할 조치를 함께 적었다.

핵심 결론 먼저:

- **기술적으로 가능하지만, 모든 계층에 하드 리밋과 실패 모드가 있다.** 아무 방어 없이 켜면 (a) 계정 제재, (b) 법 위반 과태료, (c) 예상보다 큰 서버비를 부른다.
- **가장 큰 비용은 저장소가 아니라 Inngest 실행량과 Supabase 컴퓨트 애드온**이다. 저장소는 보존정책만 있으면 관리된다.
- **안전한 사용 레인은 "키워드로 사용자가 먼저 요청한 정보성 응답"**이다. 프로모션(광고성) 발송은 별도 수신 동의를 받은 경우로 게이팅해야 한다.
- 실제 발송 백엔드는 API-last 단계다(현재는 mock/데모). 아래 TODO는 그 단계에서 반드시 구현한다.

## 1. 비용 분석

### 시나리오 가정
- 대행사 1곳 = 관리 인스타 계정 20개
- 활성 게시물 계정당 하루 약 500댓글 → **하루 약 1만 댓글/대행사** (월 약 30만)
- 키워드 매칭률 약 20% → 하루 약 2,000건 자동 DM
- 기준 요금: Supabase Pro $25/월(DB 8GB, 파일 100GB, 컴퓨트 크레딧 $10), Inngest Hobby 무료 5만 실행/월·Pro $75/월 100만 실행

### 1-1. Inngest 실행량 — 진짜 비용 동인 #1 (high)
- Inngest는 **실행 = 함수 run + 각 step.run()** 으로 과금.
- 댓글마다 함수(dedupe·send·log 3스텝)를 돌리면 4실행 × 1만 = **하루 4만 실행 → 월 120만**. **대행사 한 곳이 무료 5만을 약 1.25일 만에 소진**하고 Pro 100만도 넘긴다.
- 매칭된 것만(20%) 처리해도 월 약 24만 실행/대행사 = 무료 티어의 5배.
- **조치**: 댓글마다 함수를 돌리지 말 것. 웹훅 라우트에서 서명검증 + dedupe upsert를 인라인으로 끝내고, **트리거에 매칭된 댓글만** Inngest로 큐잉. 스텝은 2개 이하로. 플랜 한도 초과분은 큐잉 자체를 안 해 실행 0. 활성 대행사가 몇 곳 넘어가면 Inngest Pro($75/월) 편성.

### 1-2. Supabase 컴퓨트 애드온 — 진짜 비용 동인 #2 (high)
- 기본 Micro(1GB RAM)는 $10 크레딧으로 커버되지만, 바이럴 게시물은 초당 수백 웹훅을 쏟아내고, Vercel 서버리스는 짧은 커넥션을 대량 생성하며, 고빈도 로그 테이블은 autovacuum을 무겁게 만든다.
- 활성 계정이 늘면 Small($15) → Medium($60)로 올라가고, **이 애드온이 $25 기본요금·저장 초과보다 먼저 최대 지출 항목이 된다.**
- **조치**: 서버리스는 반드시 Supavisor 트랜잭션 풀러 경유(직접 커넥션 금지). 웹훅 라우트는 단일 빠른 upsert만. 로그는 DELETE가 아니라 파티션 DROP으로 정리해 autovacuum을 임계경로에서 뺀다. 컴퓨트는 피크가 아니라 지속 쓰기율 기준으로 사이징.

### 1-3. 저장소 — 보존정책이 관건 (high)
- 계정당/일: 웹훅 로그 약 10MB + dedupe 약 3MB + 발송 로그 약 0.7MB + 동의 약 0.3MB = **약 14MB/일 → 약 420MB/월**.
- 대행사 20곳이면 **월 약 8.4GB — Pro 포함 8GB를 매달 새로 채운다.** 1년 누적 약 100GB면 초과분 약 92GB × $0.125 = 월 약 $11.5 + 프로비저닝 디스크.
- **원본 웹훅 페이로드가 전체 저장의 약 70%.** 원본을 장기 보관하지 말고 추출 필드(comment_id, media_id, from_id, verb, ts)만 약 200B로 저장하면 하루 10MB→2MB.
- **조치**: 30~90일 TTL + 월별 파티션 + 오래된 파티션 DROP. 보존정책만 지키면 정상상태 약 1~2GB로 유지돼 초과 거의 없음. 8GB 도달 전 알림.

### 1-4. 비용이 아닌 것
- 인바운드 웹훅(무료 인그레스), 텍스트 템플릿(파일 저장 ~0), 댓글 작성자는 우리 Auth 사용자가 아님(10만 MAU 무관), rule-list 조회는 250GB 이그레스 근처도 안 감. **DM 템플릿에 이미지 카드를 넣기 시작하면** 파일 저장·이그레스를 다시 봐야 한다(현재는 텍스트/참조 기반 유지 권장).

### 1-5. 월 예상 비용 (개략)
| 규모 | Supabase | Inngest | 합계(개략) |
|---|---|---|---|
| 초기(활성 계정 소수) | $25 (Micro, 크레딧 내) | $0 (무료 티어) | 약 $25/월 |
| 성장(대행사 2~3) | $25 + Small/Medium 애드온 $5~50 | $75 (Pro) | 약 $105~150/월 |
| 확장(대행사 20+, 보존정책 有) | $25 + 컴퓨트 + 저장 초과 $10~ | $75 + 사용량 초과 | 수백 $/월 (보존정책 없으면 급증) |

> 요점: **보존정책·매칭시에만 큐잉·풀러**를 처음부터 넣으면 비용은 선형적으로 관리된다. 안 넣으면 대행사 1곳만 붙어도 무료 티어가 하루 만에 터진다.

## 3. 정책 · 법률 리스크

### 3-1. Meta 스팸 정책 → 사용자 본계정 제재 (high)
- Private Reply는 스팸에 민감한 좁은 채널. 동일 프로모 템플릿을 다수에게 = 안티스팸이 겨냥하는 바로 그 패턴(동일 문자열 반복·링크 밀도·수신자 불일치·신고량). 초과 시 24~48h 차단 → 장기·영구 정지.
- 공식 API 도구도 잔여 밴 리스크(분기 약 0.4%) 존재. **월 발송 한도를 플랫폼 현실보다 높게 팔면 사용자에게 계정 정지행 급행권을 파는 셈.**
- **조치**: 수신자당 24h 1회 강제, 자체 스로틀을 상한 아래로, 신고/차단 신호 급증 시 규칙 자동 일시정지(서킷 브레이커). 플랜 한도는 플랫폼 현실 **아래로**. UI에서 "한도는 과금이 아니라 안전장치"임을 고지.

### 3-2. 정보통신망법 제50조 — 광고성 DM은 명시적 사전 동의 필요 (high)
- 영리목적 광고성 정보는 **수신자의 명시적 사전 동의** 필요. **댓글은 동의가 아니다.** 거래관계 예외도 성립 안 함(댓글 작성자는 거래로 연락처를 수집한 고객이 아님). 위반 시 **과태료 최대 3,000만 원**, "전송하게 한 자"도 부과 대상.
- **조치**: 템플릿을 **정보성(사용자가 키워드로 요청한 링크 전달)** 과 **광고성(프로모·할인)** 으로 분리. 정보성 키워드-요청 플로우는 자유롭게, 광고성은 **별도 광고 수신 동의 근거**로 게이팅하고 UI 경고. 광고 문구 린터로 미동의 시 저장 차단. 광고성 분류 경계는 CLAUDE.md 규칙대로 TODO(사람 법률 검토).

### 3-3. 광고성 메시지 필수 표시 요건 (high)
- 합법 광고성 정보는 **맨 앞 (광고) 표기, 전송자 명칭·연락처, 끝에 무료 수신거부 방법**이 필수. 추가로 **21:00~08:00 야간 발송은 별도 야간 수신 동의** 필요.
- **조치**: 광고성 템플릿 에디터가 (광고) 접두 + 전송자 정보 + 무료 수신거부 안내를 자동 삽입·검증(사용자가 지울 수 없게). 수신거부/차단 리스트로 옵트아웃 영구 반영(무료). 야간 발송은 차단하거나 야간 동의 요구. **현재 구현**: 광고성 토글 시 (광고)·수신거부 문구를 본문 끝에 자동 삽입(에디터 미리보기에 반영). 전송자 정보·야간 홀드는 백엔드 TODO.

### 3-4. 개인정보보호법 — 댓글 핸들·내용·DM은 개인정보 (high)
- 작성자 @핸들, 댓글 내용, 키워드 매칭, DM 스레드 저장은 개인정보 처리. 처리방침에 적는 것만으로는 동의가 아님 — 적법 근거·목적 제한·보존기간·제3자/처리 고지 필요. 이들은 **제3자(계정주의 오디언스)** 라 동의 사슬은 계정주(컨트롤러)에게, 뷰스코프/핀치는 최소수집·안전보호 의무.
- **조치**: 규칙에 필요한 것만 저장 + 보존기간 + 핸들/메시지 로그 자동 파기(예: N일). 스키마/접근제어에서 핸들·메시지를 개인정보로 취급. **지금 mock 단계에 필드·보존을 설계에 넣어 백엔드가 상속하게.** 처리 목적·보존기간을 처리방침에 명시.

### 3-5. 수탁자(처리자) — 개인정보보호법 제26조 위탁계약 (medium)
- 계정주를 대신해 DM 발송·데이터 저장 시 핀치는 **수탁자**. 서면 위탁계약(목적제한·안전조치·재위탁 제한·감독·책임) 필요, 계정주는 자기 처리방침에 핀치를 수탁자로 고지. 데이터가 Meta(미국) 경유 → 국외이전 고지.
- **조치**: 연결/온보딩 시 인앱 DPA(처리위탁계약) 동의 단계. 계정주가 자기 처리방침에 붙일 수탁자 고지 스니펫 제공. 국외이전 고지. (법률 검토 대상 문안)

### 3-6. 책임 배분 — 계정주가 전송자/컨트롤러 (high)
- 법적으로 계정주가 광고 전송자·개인정보처리자. 그러나 UI가 "댓글 단 모두에게 프로모 발사"를 암시하면 핀치가 "전송하게 한 자"(역시 3,000만 원)로 보일 위험 + 밴 사태의 평판 부담.
- **조치**: 온보딩에서 "본인이 광고 전송자로서 동의·수신거부 책임을 지며 적법 근거를 보유함" 확인. 규칙별 고지. 무차별 프로모를 마케팅/기본값으로 두지 않음. 컴플라이언스 레일(린터·스로틀·옵트아웃)로 **기본 경로가 곧 합법 경로**가 되게.

## 5. 현재 구현이 이미 반영한 안전장치

- **Instagram 전용 고정**: 사이드바→앱 페이지 진입, 상단 채널 인디케이터, 규칙 UI, 마케팅 페이지 모두 'Instagram 전용 + 스레드/틱톡 DM API 없음'을 명시.
- **웹훅 보안**: `app/api/webhooks/instagram/route.ts` — 구독 검증 핸드셰이크 + **원문 기준 HMAC-SHA256 서명검증(타이밍 세이프)** 통과 전 무처리. 시크릿은 서버 전용(NEXT_PUBLIC 금지).
- **광고성 표기 이중 방어**: 에디터 토글 시 (광고)·수신거부 문구 자동 삽입 + 미리보기 반영, **서버 액션(`actions.ts`)에서도 `enforceAdDisclosure`로 재강제**.
- **하루 발송 상한(dailyCap)**: 규칙별 캡 + 초과 시 '상한 도달' 표시. 무차별 발송 방지의 첫 관문.
- **1회/7일 원칙을 UI 카피로 반영**: 공개 답글은 Private Reply 1회 제한을 보완하는 수단으로 설명.
- **정직한 연동 상태 고지**: 실제 발송은 메타 심사 후 활성화됨을 앱 상단에 명시(과대 약속 회피).
- **발송 성공률·응답률은 자체 추정치가 아님**(메타 실제 결과)임을 InfoTip으로 구분 고지.
- **로그인 페이지 크롤 차단**: `robots.ts`에 `/auto-dm` 추가(마케팅 페이지 `/instagram/auto-dm`는 색인 유지).
- **인증 재확인**: 서버 액션이 레이아웃 가드에만 의존하지 않고 자체 `getUser()` 재확인 + 소유권 검증 TODO.

## 7. 결론 · 권고

- **되는 기능이다. 단, 컴플라이언스 레일을 1급 기능으로 함께 출시**해야 안전하다(린터·스로틀·옵트아웃·동의 게이팅·상태 리포트).
- **안전한 레인**: 키워드로 사용자가 먼저 요청한 **정보성 응답**(요청한 링크 전달). **광고성 발송은 별도 수신 동의를 받은 대상**으로만.
- **마케팅 문구 주의**: '무제한/대량 자동 발송'류 금지. 플랫폼 한도(약 200/시, 1인 24h 1회)를 왜곡하지 않는다.
- **비용**: 보존정책 + 매칭시 큐잉 + 커넥션 풀러를 처음부터. 이것만 지키면 초기 약 $25/월, 성장기 약 $100~150/월 수준으로 관리 가능.
- 확신이 필요한 법적 판단(광고성 분류, 청소년 대상, 개인정보 보존기간, DPA 문안)은 코드에 TODO로 남겼고, **사람(법률 검토)** 이 확정할 영역이다.
