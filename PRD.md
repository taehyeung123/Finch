## PART 1. 프로젝트 개요

### 1.1 프로젝트 정의

인스타그램·틱톡·쓰레드(Threads) 3개 SNS 채널의 데이터를 한곳에서 분석하고, 메타광고 집행까지 관리할 수 있는 통합 SaaS 플랫폼입니다. SNS를 운영하는 개인·브랜드, 메타광고를 집행하는 광고주, 이들을 대행하는 콘텐츠 마케터·대행사가 공통으로 쓸 수 있는 데이터 도구를 지향합니다.

한 줄로 요약하면:

> **"내 채널 현황부터 경쟁사 동향, 트렌드 파악, 광고 집행, 콘텐츠 제작까지 — SNS 마케팅에 필요한 모든 데이터 작업을 하나의 대시보드에서."**

### 1.2 핵심 가치제안 (Value Proposition)

| 사용자 그룹 | 핵심 문제(Pain Point) | 이 서비스가 주는 가치 |
|---|---|---|
| SNS 크리에이터 / 브랜드 운영자 | 앱을 오가며 채널 성과를 따로 확인, 콘텐츠 기획을 감(感)에 의존 | 3채널 통합 대시보드 + 데이터 기반 콘텐츠 기획(트렌드/카테고리 탐색) |
| 메타광고 광고주 | 광고 성과와 오가닉 콘텐츠 성과를 따로 봐야 함, 경쟁사 동향 파악 어려움 | 광고+오가닉 통합 뷰, 경쟁사 광고 자동 모니터링 |
| 콘텐츠 마케터 / 대행사 운영자 | 클라이언트 보고서 수작업 취합, 콘텐츠 아이디어 발굴에 많은 시간 소요 | 자동 리포트, AI 콘텐츠 아이디어·카드뉴스·숏폼 생성 |

### 1.3 대상 채널 & 사용자 범위

- **연동 채널**: Instagram · TikTok · Threads (3채널) + Meta 광고 계정(광고 관리 목적)
- **핵심 사용자 3그룹**: ① 개인/브랜드 크리에이터 ② 메타광고 광고주 ③ 콘텐츠 마케터·대행사
- 3개 채널은 플랫폼마다 공식적으로 제공하는 데이터 범위가 크게 다릅니다. 이 차이를 감추지 않고 기획서 전반과 실제 UI(PART 3, 6)에 명시적으로 반영하는 것이 이 프로젝트의 중요한 설계 원칙입니다.

### 1.4 페르소나

**페르소나 A — 개인 크리에이터 "민지" (28세)**
릴스와 틱톡을 오가며 숏폼 콘텐츠를 만들고 쓰레드로 소통하는 1인 크리에이터. "이번 주에 뭘 올려야 터질까"를 늘 고민하며, 카테고리별 인기 콘텐츠와 내 채널 스타일 분석을 가장 중요하게 봅니다.

**페르소나 B — 메타광고 광고주 "재현" (35세, D2C 브랜드 마케팅 담당)**
매달 메타광고 예산을 직접 운용하며, 광고 성과와 함께 경쟁 브랜드가 어떤 소재로 광고하는지를 상시 체크하고 싶어 합니다. 대시보드 하나로 광고 계정과 채널 현황을 같이 보고 싶어 합니다.

**페르소나 C — 대행사 마케터 "소연" (31세, 콘텐츠 마케팅 에이전시 AE)**
여러 클라이언트 계정을 동시에 관리하며, 매달 성과 보고서를 만드는 데 많은 시간을 씁니다. 콘텐츠 아이디어 발굴과 카드뉴스 제작을 빠르게 처리할 도구가 필요합니다.

### 1.5 벤치마크 — 블링(vling)에서 무엇을 가져오고, 무엇을 새로 정의하는가

조사 결과 **"블링(vling)"은 유튜브 인플루언서 마케팅에 특화된 분석 플랫폼**입니다(일부 페이스북·인스타그램·틱톡 비교 기능이 언급되긴 하나 핵심은 유튜브이며, "국내 1위 유튜브 분석 플랫폼"을 표방합니다). 확인된 주요 기능은 다음과 같습니다.

- 채널 검색·분석 / 영상 검색·분석 / 시청자 분석
- 광고단가 확인 · 채널 비교 · **AI가 자동 작성하는 광고 제안서**
- 카테고리별 랭킹 + 자체 "알고리즘 스코어"(영상이 알고리즘을 얼마나 타는지 수치화)
- "요즘 뜨는 채널" 실시간 확인 / 관심 채널의 업로드·성과 변화 실시간 추적(모니터링)
- 유튜브 수익 계산기, **AI 콘텐츠 메이커**(키워드 경쟁력 + 스크립트 초안 제공 — 영상 자체를 생성하진 않음)
- **AI 유튜버 파인더**(상품 URL/키워드 입력 → 적합 채널 추천 → 광고 제안서 자동 작성 → 메일 발송)
- 댓글 긍정/부정 감성 분석 + 영상별 Top 10 댓글, 엑셀/CSV 추출
- 무료~유료 등급형 멤버십(자동 업그레이드 지원, 다운그레이드는 해지 후 재가입 방식)

**경쟁 구도 참고**: 블링과 함께 언급되는 서비스로 Modash(3.5억 개 글로벌 IG/틱톡/유튜브 DB, 해외 중소 브랜드용), 태그바이(TAGby, 캠페인 운영 자동화+정산까지 원스톱), 레뷰, 피처링 등이 있습니다. 공통적으로 "데이터 분석 → 매칭/캠페인 관리 → 정산/리포트"로 갈수록 차별화가 강해지는 구조입니다.

**이 프로젝트가 차용할 구조**
- 검색 → 분석 → 비교의 3단 정보 구조
- "실시간 모니터링"이라는 개념 자체(관심 계정/키워드 등록 → 변화 감지 → 알림)
- 카테고리 랭킹 + 자체 스코어링 방식
- AI가 초안(스크립트/제안서/카피)을 만들어주는 어시스턴트형 UX
- 등급형 요금제(자동 업그레이드)

**이 프로젝트가 새로 정의해야 하는 부분**
블링은 유튜브 단일 채널이라 상대적으로 공개 데이터 접근이 자유로운 유튜브 Data API를 기반으로 폭넓은 기능을 구현할 수 있었습니다. 반면 이 프로젝트가 다루는 인스타그램·틱톡·쓰레드는 공식 API가 철저히 **"내 계정" 중심**으로 설계되어 있어, 동일한 이름의 기능이라도 채널별로 실제 구현 가능한 정도가 크게 다릅니다. 이 격차를 투명하게 설계에 반영하는 것이 PART 2의 핵심 내용이며, 이 서비스만의 신뢰도로 이어질 수 있는 부분이라고 판단됩니다.

## PART 2. 기술 실현성 검토 ⚠️ 개발 착수 전 반드시 확인

요청 주신 기능 목록에는 명시적으로 "확인해달라"고 하신 항목(메타광고 컨트롤, AI 영상 제작)과, 별도 확인 요청은 없었지만 실제로는 공식 API만으로는 안 되는 기능(타계정 분석, 실시간 트렌드, 카테고리 탐색)이 섞여 있습니다. Claude Code로 실제 개발에 들어가기 전에 이 부분부터 명확히 짚어야 이후 설계·일정·비용 산정이 전부 틀어지지 않기 때문에, 이 기획서에서 가장 먼저·가장 비중 있게 다룹니다.

### 2.1 결론 요약 (바쁘시면 이 표만 보셔도 됩니다)

| 요청하신 기능 | 공식 API만으로 가능? | 결론 |
|---|---|---|
| 메타광고 컨트롤(조회·생성·운영) | ✅ 가능 | Meta Marketing API로 캠페인 생성·관리·최적화까지 실제로 가능. 단 앱 심사·비즈니스 인증 필요 |
| 경쟁사 광고 자동 수집·알림 | ✅ 가능 — 사실상 **가장 실현하기 쉬운 기능** | Meta 광고 라이브러리(Ad Library) API가 정확히 이 용도로 존재. 무료·공식 |
| 내 계정(IG/TikTok/Threads) 대시보드, 조회수·게시물·댓글 | ✅ 가능 | 3채널 모두 공식 API로 완전 지원 |
| 카드뉴스 AI 생성 | ✅ 가능 | 텍스트 생성(예: Claude API) + 템플릿 렌더링 조합. 이미 상용화되어 있는 구조 |
| 숏폼 영상 AI 생성 | ✅ 조건부 가능 | 외부 생성형 AI 영상 API 연동으로 가능. 실사 촬영을 완전히 대체하는 수준은 아직 아님 |
| URL 입력 → 타 계정 영상 분석 | ⚠️ 부분 가능 | 인스타그램은 계정명을 정확히 알아야 기초 정보만 조회 가능(Business Discovery). 틱톡·쓰레드는 공식 API로는 사실상 불가 |
| 카테고리별 탐색 / 실시간 트렌드 | ⚠️ 공식 API만으로는 불가 | 3채널 모두 "플랫폼 전체 트렌드 검색"을 공식 API로 지원하지 않음. 3rd party 데이터 공급자 연동이 사실상 필수 |
| 경쟁 타계정 정밀 분석(시청자층 등) | ⚠️ 공식 API만으로는 불가 | Business Discovery는 기초 지표만 제공. 정밀 분석에는 3rd party 필요 |

**한 문장 결론**: *"내 계정"에 관한 기능은 3채널 모두 공식 API로 100% 커버됩니다. 반대로 "타 계정 / 트렌드 / 카테고리 탐색"에 관한 기능은 3채널 모두 공식 API의 설계 목적 자체가 아니어서, 유료 데이터 공급업체(3rd party)를 붙여야 실제로 작동합니다.* 이건 이 프로젝트만의 특수한 제약이 아니라 Socialinsider·Modash·EchoTik 같은 상용 SNS 분석 플랫폼들도 전부 "공식 API + 유료 데이터 공급업체" 하이브리드 구조로 만들어져 있습니다.

### 2.2 채널별 공식 API 상세

**Instagram Graph API (Meta, 무료)**

- 비즈니스/크리에이터 계정만 지원됩니다. 개인 계정은 2024년 12월 Basic Display API가 완전히 폐지되어 접근이 불가능합니다 → **온보딩 과정에서 "비즈니스/크리에이터 계정 전환"을 필수 안내 단계로 넣어야 합니다.**
- 내 계정 기준으로는 게시물 발행(사진/영상/릴스/캐러셀/스토리), 인사이트(도달·노출·참여·팔로워 인구통계), 댓글 관리, DM(수신 후 24시간 내 응답 규칙)까지 전부 지원됩니다.
- **Business Discovery 엔드포인트**: 다른 비즈니스/크리에이터 계정의 팔로워 수·게시물 수·bio·최근 게시물(좋아요/댓글 수 포함)을 "정확한 계정명"으로 조회할 수 있습니다. 상대방의 별도 동의가 필요 없습니다. 단 팔로워 목록·인구통계 데이터는 제공되지 않습니다.
- **해시태그 검색**: 계정당 주(7일 롤링) 30개 해시태그까지만 조회 가능하며, top/recent 게시물을 볼 수 있으나 recent는 최근 24시간 이내로 제한되고 게시자 계정명은 응답에 포함되지 않습니다.
- Rate limit: 앱당 시간당 200회 호출.

→ "타계정 분석"은 계정명을 아는 소수 경쟁사를 등록해 기초 지표를 보는 용도로는 공식 API만으로 충분하지만, "카테고리로 검색해서 새로운 계정을 발견하기"나 "실시간 트렌드 전체 보기"는 주 30개 해시태그 제한 때문에 실서비스로 쓰기엔 명백히 부족합니다.

**TikTok API (TikTok for Developers)**

- Login Kit / Content Posting API / Display API: 사용자가 로그인 연동한 **본인 계정**의 게시·조회·기본 프로필 정보 중심으로 설계되어 있습니다.
- **Research API**: 미국·유럽의 비영리 대학 소속 연구자만 신청 가능하며 상업적 이용이 명시적으로 금지되어 있습니다. 일 1,000건 제한, 데이터도 최대 7일까지 지연됩니다.
- Business/Shop API: 광고 리포팅·이커머스 중심이며 이 역시 본인(또는 위임받은) 계정 범위입니다.

→ 3채널 중 **틱톡이 공식 API로 커버 가능한 범위가 가장 좁습니다.** "내 계정" 기능(연동, 내 게시물 조회, 예약 발행)은 문제없지만, 타계정 분석·카테고리 탐색·실시간 트렌드는 공식 경로가 사실상 없어 처음부터 3rd party 데이터 공급자를 전제로 설계해야 합니다. 이 부분은 서비스 내에서도 "틱톡은 인스타그램보다 제공 가능한 기능이 제한적일 수 있다"고 투명하게 안내하는 UX를 권장드립니다(PART 3, PART 6 참고).

**Threads API (Meta, 무료)**

- 2024년 6월 공개 API가 시작되었고, 게시·읽기·인사이트·답글 관리·삭제가 지원됩니다.
- Rate limit: 24시간당 게시 250건 / 답글 1,000건 / 삭제 100건, 노출수 기반 산식이 함께 적용됩니다.
- 공식 문서상 "전역 검색이나 폭넓은 발견(discovery) 기능은 본인 계정 게시와는 다른 영역"이라고 명시되어 있으며, 키워드 검색·프로필 발견 관련 권한(scope)이 존재하긴 하나 여전히 자기 계정 생태계 중심으로 설계되어 있습니다.

→ 쓰레드는 셋 중 가장 최근에 API가 열린 채널이라 계속 확장되는 중이지만, 현재로선 "내 계정" 기능 중심으로 설계하는 것이 안전합니다.

**Meta Marketing API (광고 관리, 무료)**

- **Standard Access**: 본인 광고 계정만 접근 가능하며 앱 심사가 필요 없습니다 → 광고주가 자기 광고 계정을 붙이는 MVP 단계는 이것만으로 충분합니다.
- **Advanced Access**: 클라이언트(제3자) 광고 계정을 다루려면 필요하며, **비즈니스 인증(사업자등록증 제출) + 앱 심사**를 통과해야 합니다. 대행사향 기능을 넣으려면 이 심사를 반드시 통과해야 하므로 일정에 미리 반영해야 합니다.
- 캠페인·광고세트·소재의 생성/수정/일시정지 등 실제 "컨트롤"이 API로 전부 가능합니다. Revealbot, AdEspresso 같은 상용 광고관리 SaaS들이 실제로 이 방식으로 동작합니다.
- 2026년 기준 Advantage+ 체계로 계속 통합되는 중이며, Meta가 최근(2026년 4월) 서드파티 AI 도구가 광고 생태계에 더 깊이 연동되도록 "Meta ads AI connectors"를 공개해 AI 추천·자동화 기능과의 결합에 우호적인 방향으로 가고 있습니다.

**Meta 광고 라이브러리(Ad Library) API — "경쟁사 광고 자동 모니터링"의 정답 ⭐**

- Meta가 투명성 목적으로 운영하는 **완전 무료·공개 API**(`/ads_archive` 엔드포인트)입니다.
- 로그인 없이도 누구나 브랜드명·키워드·국가·광고유형으로 검색할 수 있으며, API로는 광고 소재(이미지/영상 URL), 노출 플랫폼(FB/IG), 게재 기간, 노출수 구간 등을 가져올 수 있습니다.
- 단, **한국은 광고비(스펜드) 데이터가 비공개**이며(해외 일부 지역만 지출액 공개), 원칙적으로 **현재 온에어 중인 활성 광고만** 조회할 수 있습니다(정치·사회이슈 광고는 예외적으로 과거 이력 보관).
- "새 광고 알림" 기능 자체를 API가 제공하지는 않으므로, **주기적으로(예: 하루 1~4회) 등록된 경쟁사 페이지를 폴링 → 이전 스냅샷과 비교(diff) → 신규 광고 감지 시 알림**을 직접 구현해야 합니다. 말씀하신 대로 이건 실제 AI가 필요한 일이 아니라 스케줄러 + DB diff만으로 충분히 구현되는 백엔드 로직입니다.

→ 전체 요청 기능을 통틀어 **가장 안정적이고 가장 "공식적인" 방법으로 구현할 수 있는 기능이 바로 이 "경쟁사 광고 모니터링"**입니다. 카테고리 트렌드나 타계정 정밀 분석보다 오히려 개발 우선순위를 앞당겨도 좋을 만큼 리스크가 낮은 기능입니다.

### 2.3 데이터 갭을 메우는 방법 — 3rd party 데이터 공급자

"타계정 분석 / 카테고리 탐색 / 실시간 트렌드"처럼 공식 API만으로 부족한 기능은, 시중 SNS 분석 서비스 대부분이 그렇듯 **유료 데이터 공급업체를 API로 추가 연동**해서 채웁니다. 조사한 주요 옵션은 다음과 같습니다.

| 공급자 | 커버 플랫폼 | 특징 | 대략적 비용 |
|---|---|---|---|
| EnsembleData | TikTok, Instagram, YouTube 등 8개 | 인플루언서 마케팅·소셜리스닝 업체들이 많이 사용, 유닛 기반 과금 | 월 $100~1,400대 |
| Phyllo | 20개+ 플랫폼 | **크리에이터 본인 동의(OAuth) 기반** — 공식 연동에 가까워 정책 리스크가 낮음. 단 상대가 동의해야 하므로 "비동의 경쟁사 계정" 분석엔 쓸 수 없음 | 월 $199~ |
| HikerAPI | 인스타그램 전용 | 요청당 과금이라 소량 테스트에 적합, 엔드포인트가 다양(팔로워/해시태그/위치 등) | 요청당 $0.0006 |
| Modash | Instagram / TikTok / YouTube | 인플루언서 발굴·캠페인 관리에 특화 | 별도 문의 |
| Apify | IG/TikTok/FB 등 (마켓플레이스형) | 커뮤니티가 만든 스크래퍼 조합, 유연하지만 스키마 일관성이 낮고 계정정지 리스크 있음 | 컴퓨팅 단위 과금 |
| Bright Data | 8개 플랫폼 | 엔터프라이즈급, 프록시 인프라가 강력 | 월 $500~(고가) |

**권장 방향**: MVP 단계에서는 "내 계정" 기능(공식 API)과 "경쟁사 광고 모니터링"(Meta Ad Library, 공식 API)만으로도 충분히 서비스를 낼 수 있습니다. "타계정 분석 / 카테고리 탐색 / 실시간 트렌드"는 Phase 2 이후, 예산 규모에 맞춰 HikerAPI(저비용 테스트) → EnsembleData·Modash(본격 운영) 순으로 단계적으로 붙이는 것을 제안합니다. 이 비용은 원가 구조에 직접 영향을 주므로 요금제 설계(PART 9)와 함께 검토가 필요합니다.

### 2.4 AI 콘텐츠 생성 실현성

**카드뉴스 생성 — 가능합니다.** "주제 입력 → 슬라이드별 텍스트/카피 생성(LLM, 예: Claude API) → 정해진 템플릿에 자동 배치(이미지 렌더링)"의 3단 조합으로, 이미 여러 상용 서비스가 이 구조로 운영되고 있습니다. 브랜드 톤·폰트를 미리 저장해두면 일관된 스타일로 반복 생성할 수 있습니다.

**숏폼 영상 생성 — 조건부 가능합니다.** 두 가지 접근 방식이 있고 성격이 다릅니다.

1. **템플릿 기반 모션그래픽**: 제품 사진+카피만 넣으면 정해진 템플릿(자막 위치, 전환 효과, BGM)에 자동 합성됩니다. 브랜드 일관성이 높고 비용이 예측 가능해 "매주 광고 소재 찍어내기"에 적합합니다.
2. **생성형 AI 영상**(Sora 2, PixVerse, Seedance 2.0/Kling 등): 텍스트나 이미지 한 장으로 영상을 생성합니다. 자유도는 높지만 매번 결과물이 달라 브랜드 일관성을 담보하기 어렵고 비용도 변동적입니다.

2026년 기준 품질 수준은 "30초 SNS 광고 영상을 5분 내 생성, 완성도는 사람이 만든 것의 70% 수준" 정도로 평가되고 있어 **SNS 광고·숏폼 용도로는 실용적이지만, 영화·CF 수준의 완성도를 기대하긴 이릅니다.** 자체 영상 생성 엔진을 밑바닥부터 만들 필요는 없고, 위와 같은 외부 API를 "연동"하는 방식이 합리적이며, Phase 3 이후 기능으로 배치하는 것을 권장합니다.

### 2.5 법적·정책 리스크 요약 (상세는 PART 12)

- 공개 데이터라도 국내 개인정보보호법상 개인 식별정보 처리에는 별도 검토가 필요합니다 — 정식 서비스 전 법률 자문을 권장합니다.
- 3rd party 스크래핑 기반 데이터는 각 플랫폼 이용약관과 충돌할 가능성이 있어 "공식 API 우선, 스크래핑은 최후 수단" 원칙을 권장합니다.
- AI 생성 콘텐츠에 대해 각 플랫폼이 "AI 생성물임을 표시"하도록 요구하는 흐름이 강화되고 있어(예: 일부 플랫폼의 AI 생성 표시 의무화), 카드뉴스·숏폼 생성 기능에도 이를 반영해야 합니다.

## PART 4. 핵심 기능 명세서

요청하신 기능(1~9번)에 더해, 요청 범위 밖이지만 필요하다고 판단되는 기능(10~13번, "**[추가 제안]**" 표기)까지 포함했습니다.

### 4.1 대시보드 (홈)

- **목적**: 로그인 직후 첫 화면. 연동한 모든 채널의 핵심 지표를 한눈에 보여줍니다.
- **핵심 UI 요소**
  - 채널 스위처(Instagram / TikTok / Threads / 전체보기) — 탭 또는 드롭다운
  - 요약 카드: 팔로워 수(증감 포함), 이번 주 조회수, 게시물 수, 평균 좋아요/댓글, 참여율
  - 최근 게시물 목록(썸네일 + 핵심 지표 미니 차트)
  - 채널별 스타일 분석 요약(예: 최근 콘텐츠 유형 비중 — 릴스/피드/스토리)
  - 광고 계정 연동 시: 이번 달 광고비·성과 요약 카드(오가닉과 나란히 배치)
- **데이터/구현**: 공식 API(각 채널 Insights) — 실시간이 아니라 배치(예: 1일 4회) 수집 후 캐싱하는 구조를 권장합니다. API 호출 제한(시간당 200회 등)을 고려할 때 사용자가 새로고침할 때마다 즉시 API를 부르는 구조는 곧 한도를 초과합니다.
- **MVP 포함**: ✅ Phase 1

### 4.2 계정 연동 관리 (설정)

- **목적**: 내 SNS 계정·광고 계정을 연동/해제하는 설정 화면.
- **핵심 UI 요소**
  - 채널별 연동 카드(Instagram/TikTok/Threads/Meta 광고 계정) — 연동 상태, 재연동(토큰 만료 시), 해제 버튼
  - 비즈니스/크리에이터 계정 전환 안내(인스타그램 개인 계정으로 연동 시도 시 안내 배너)
  - 권한(scope) 목록 표시 — 이 앱이 무엇에 접근하는지 투명하게 안내
- **데이터/구현**: OAuth 2.0, 장기 토큰(60일) 자동 갱신 스케줄러 필요
- **MVP 포함**: ✅ Phase 1 (전체 기능의 전제 조건)

### 4.3 콘텐츠 분석 (URL 입력 분석)

- **목적**: 특정 게시물/영상의 URL을 입력하면 상세 데이터를 분석해주는 기능.
- **핵심 UI 요소**
  - URL 입력창 + "분석하기" 버튼
  - 결과 화면: 조회수/좋아요/댓글/공유 추이, 업로드 후 시간대별 증가 속도 그래프, 썸네일·캡션·해시태그 분석, (가능한 경우) 댓글 감성 요약
  - "내 계정 게시물"과 "타 계정 게시물"을 구분해 표시 — 후자는 조회 가능한 지표가 더 제한적임을 명시
- **데이터/구현**: 내 게시물은 공식 API로 전체 지표, 타계정 게시물은 채널별로 상이(PART 3 매트릭스 참조) — 인스타그램은 Business Discovery, 틱톡/쓰레드는 3rd party 필요
- **MVP 포함**: ✅ Phase 1 (내 계정 한정) → 🔶 Phase 2 (타계정까지 확장)

### 4.4 카테고리별 탐색 & 실시간 트렌드

- **목적**: 분야(뷰티, 푸드, 패션 등)를 골라 요즘 뜨는 콘텐츠를 발견하는 기능. 요청 사항 중 가장 강조하신 기능입니다.
- **핵심 UI 요소**
  - 카테고리 선택(칩 형태) + 채널 필터 + 정렬(조회수순/급상승순/좋아요순)
  - 콘텐츠 카드 그리드: 썸네일, 채널 배지, 핵심 지표, "팔로워 대비 조회수 비율" 같은 블링식 스코어링(구독자 기준인 블링 스코어링을 팔로워 기준으로 재해석)
  - "실시간" 탭과 "카테고리" 탭 분리, 각 카드에 "저장" 버튼(관심 리스트에 담기 → 4.5 타계정 분석으로 연결)
- **데이터/구현**: 🔶 3rd party 데이터 공급자 필수(PART 2.3 참고). 데이터 최신성은 공급자의 갱신 주기에 종속되므로 "n분 전 기준" 같은 타임스탬프 표기를 반드시 넣어야 합니다.
- **MVP 포함**: 🔶 Phase 2 (3rd party 계약 이후)

### 4.5 타계정(경쟁사) 분석

- **목적**: 특정 계정을 등록해 지표를 추적하고, 여러 계정을 비교하는 기능.
- **핵심 UI 요소**
  - 계정 검색/등록(정확한 사용자명 입력 또는 4.4에서 저장한 계정 불러오기)
  - 프로필 요약: 팔로워·게시물 수·평균 참여율(공식 API 범위 내)
  - 채널별 "분석 가능 범위" 배지 — 인스타그램은 기초 지표, 틱톡·쓰레드는 "제휴 데이터 공급사 데이터"임을 명시
  - 2~3개 계정 나란히 비교하는 비교 뷰(업로드 주기, 콘텐츠 길이, 평균 조회수 등)
- **데이터/구현**: 인스타그램 Business Discovery(공식) + 3rd party(정밀 분석)
- **MVP 포함**: ⚠️ Phase 2 (인스타그램 기초 버전) → 🔶 Phase 3 (3채널 정밀 버전)

### 4.6 경쟁사 광고 자동 모니터링

- **목적**: 경쟁사를 등록해두면 새로 집행되는 메타광고를 자동으로 수집해 알려주는 기능. PART 2.2에서 설명한 대로 **Meta 광고 라이브러리 API로 구현 가능한, 신뢰도가 가장 높은 기능**입니다.
- **핵심 UI 요소**
  - 경쟁사(페이지) 등록 리스트
  - 광고 피드: 신규 감지된 광고 소재(이미지/영상), 게재 시작일, 노출 플랫폼(FB/IG), 게재 기간(진행 중 광고는 "n일째 운영 중" 표기 — 오래 운영될수록 성과가 좋다는 신호로 해석 가능)
  - 신규 광고 발견 시 알림(인앱 + 이메일)
  - **참고**: 한국은 광고비(스펜드) 데이터가 비공개이므로 이 지표는 노출하지 않거나 "비공개" 표기
- **데이터/구현**: Meta Ad Library API(`/ads_archive`) + 스케줄러(1일 1~4회 폴링) + 이전 스냅샷 대비 diff 로직. 순수 백엔드 배치 작업으로, 말씀하신 대로 실제 AI 판단이 필요하지 않습니다.
- **MVP 포함**: ✅ Phase 1~2 권장 (구현 난이도 대비 신뢰도가 가장 높아 우선순위를 앞당길 것을 제안)

### 4.7 메타광고 매니저 연동

- **목적**: 메타광고 계정을 연동해 캠페인 현황을 보고, 나아가 직접 캠페인을 생성·관리하는 기능.
- **핵심 UI 요소**
  - 1단계(리포트): 캠페인/광고세트/소재별 성과 테이블 + 오가닉 콘텐츠와 나란히 보는 통합 뷰
  - 2단계(관리): 캠페인 생성 마법사(목표 선택 → 타겟팅 → 예산 → 소재 업로드 → 게재), 예산 수정, 일시정지/재개
  - AI 추천 배지: "이 소재의 CTR이 평균보다 낮습니다", "예산 소진 임박" 같은 규칙 기반 알림(초기엔 AI가 아니라도 무방, 추후 고도화)
- **데이터/구현**: Meta Marketing API. 1단계는 Standard Access로 충분, 2단계 중 광고주 본인 계정 운영은 Standard Access로 가능하며, **대행사가 클라이언트 계정을 관리**하려면 Advanced Access(비즈니스 인증+앱 심사) 필요
- **MVP 포함**: ⚠️ Phase 2 (리포트 조회) → 🔶 Phase 3 (캠페인 생성·관리, Advanced Access 심사 완료 후)

### 4.8 AI 콘텐츠 스튜디오 (카드뉴스 / 숏폼 / 아이디어 추천)

- **목적**: 콘텐츠 기획·제작 시간을 줄여주는 AI 생성 도구 모음.
- **핵심 UI 요소**
  - **콘텐츠 아이디어 추천**: 카테고리/키워드 입력 → 트렌드 데이터(4.4) 기반 주제 후보 리스트 + 예상 반응 근거
  - **카드뉴스 생성기**: 주제/링크 입력 → 슬라이드별 카피 자동 생성 → 브랜드 톤·폰트·컬러 템플릿 적용 → 슬라이드 단위 수정 → 이미지 내보내기
  - **숏폼 영상 생성기**: (a) 템플릿 기반(제품사진+카피 → 자동 합성) (b) 생성형 AI 영상(텍스트/이미지 → 영상) 두 모드 제공, AI 생성물에는 자동으로 워터마크/표시 부착(정책 준수)
  - 생성물은 바로 4.7 예약 발행 또는 다운로드로 연결
- **데이터/구현**: 텍스트 생성은 Claude API, 이미지 렌더링은 자체 템플릿 엔진(HTML/Canvas), 영상은 외부 생성형 AI 영상 API 연동(PiAPI/Runway 등, PART 2.4 참고)
- **MVP 포함**: ✅ Phase 2 (카드뉴스, 아이디어 추천) → 🔶 Phase 3 (숏폼 영상)

### 4.9 AI 에이전트

- **목적**: 챗 형태로 위 모든 기능을 자연어로 호출할 수 있는 대화형 어시스턴트. "이번 주 우리 인스타 어때?", "요즘 뷰티 카테고리에서 뜨는 릴스 찾아줘", "경쟁사 A 계정 분석해줘", "이 주제로 카드뉴스 만들어줘" 같은 요청을 처리합니다.
- **핵심 UI 요소**
  - 대시보드 우측 또는 하단에 상시 노출되는 채팅 패널
  - 각 응답에 실제 데이터/화면으로 바로 이동하는 링크·카드 첨부(순수 텍스트 답변이 아니라 위젯형 응답)
  - 대화 이력 저장, 자주 쓰는 질문 프리셋 제공
- **데이터/구현**: 플랫폼 내부 기능을 "도구(tool)"로 등록해 Claude API의 Function Calling으로 호출하는 구조를 권장합니다. 예: `get_dashboard_summary()`, `search_trending(category, channel)`, `analyze_account(handle, channel)`, `analyze_content(url)`, `generate_card_news(topic)`, `get_competitor_ads(competitor_id)` 등. 즉, AI 에이전트는 별도 기능이 아니라 **이미 만든 4.1~4.8 기능들의 자연어 인터페이스**로 설계하는 것이 효율적입니다.
- **MVP 포함**: 🔶 Phase 3 (기반 기능들이 먼저 갖춰진 뒤 자연어 레이어로 얹는 것을 권장)

### 4.10 팀 워크스페이스 [추가 제안]

- **목적**: 대행사·마케팅팀처럼 여러 명이 함께 쓰는 경우를 위한 협업 기능. 페르소나 C(대행사 마케터)를 위해 필요하다고 판단해 제안합니다.
- **핵심 UI 요소**: 워크스페이스 내 멤버 초대/권한(관리자·에디터·뷰어), 클라이언트별 프로젝트 분리, 클라이언트에게 특정 대시보드만 공유하는 뷰어 링크
- **MVP 포함**: 🔶 Phase 3~4 (Pro/Agency 요금제와 함께 검토)

### 4.11 리포트 자동 생성 & 내보내기 [추가 제안]

- **목적**: 블링에서도 확인된 "엑셀/CSV 추출" 패턴을 확장해, 광고주 보고용 리포트를 자동 생성하는 기능.
- **핵심 UI 요소**: 기간·채널·지표를 선택해 PDF/Excel로 내보내기, 정기 리포트 예약 발송(주간/월간), 화이트라벨(대행사 로고 삽입) 옵션
- **MVP 포함**: 🔶 Phase 3

### 4.12 알림 센터 [추가 제안]

- **목적**: 신규 트렌드, 경쟁사 신규 광고(4.6), 내 계정 이상 성장/하락 감지 등을 놓치지 않도록 모아서 보여주는 알림함.
- **핵심 UI 요소**: 인앱 알림 목록, 이메일/푸시 알림 설정, 알림 유형별 on/off 토글
- **MVP 포함**: ⚠️ Phase 2 (4.6 경쟁사 광고 알림과 함께 구축)

### 4.13 요금제 · 사용량 관리 [추가 제안]

- **목적**: 현재 사용 중인 플랜, 이번 달 사용량(콘텐츠 분석 횟수, AI 생성 횟수 등)을 보여주고 업그레이드를 유도하는 화면.
- **핵심 UI 요소**: 사용량 게이지, 플랜 비교표, 결제 수단 관리
- **데이터/구현**: PART 9(요금제 구조 제안) 참고
- **MVP 포함**: ✅ Phase 1 (무료/유료 구분이 있다면 최소 버전 필요)

### 4.14 인스타 댓글 자동 DM [추가 제안]

- **목적**: 인스타그램 게시물에 특정 조건의 댓글이 달리면 자동으로 다이렉트 메시지를 보내, 콘텐츠 마케터·광고대행사·인플루언서·쇼핑몰의 반복 응대를 자동화하고 문의를 판매·상담으로 연결한다.
- **채널 범위**: **Instagram 전용** — 스레드·틱톡은 서드파티 DM 발송 API가 없어 미지원(PART 3 매트릭스: IG=full, TikTok/Threads=none). 공개 자동 답글은 스레드에서 부분 가능하나 별개 기능.
- **핵심 UI 요소**: 게시물별 규칙 목록/생성/편집·토글, 트리거(모든 댓글 / 키워드), 발송 DM 본문·버튼, 공개 답글(선택), 광고성 토글((광고)·수신거부 자동 삽입), 하루 발송 상한, 상태별 발송 요약.
- **데이터/구현**: `AutoDmRule`/`AutoDmSummary` 타입 + 데이터 레이어(mock/empty). 실제 발송은 API-last — Instagram Messaging(Private Replies) + 댓글 웹훅. 웹훅 라우트(`app/api/webhooks/instagram`)는 HMAC-SHA256 서명검증까지 스캐폴드 완료. 서버 액션(`app/(app)/auto-dm/actions.ts`)에서 (광고) 표기·인증을 이중 방어.
- **하드 제약(반드시 반영)**: 댓글당 Private Reply **1회·7일 이내**, 계정당 레이트리밋, 토큰 60일 만료. 자동 드립/재참여 발송은 정책 위반이므로 미구현.
- **법·정책**: 정보통신망법 제50조(광고성 정보 사전 동의·(광고) 표기·수신거부·야간 발송), 개인정보보호법(댓글·핸들·메시지 = 개인정보, 수탁자 DPA). 상세는 `docs/AUTO_DM_COST_RISK.md`.
- **요금제**: 유료 전용(Creator 이상), 플랜별 월 발송 한도(PART 9). 한도는 플랫폼 안전 상한 아래로 설정.
- **MVP 포함**: 확장 제안 — UI/데이터/보안 스캐폴드는 완료, 실제 발송은 Meta 앱 심사(instagram_manage_messages + instagram_manage_comments) 이후 Phase 3+.

## PART 6. 페이지 설계

### 6.1 랜딩페이지 (반응형)

요청하신 대로 로그인 전 첫 진입 화면은 반응형 랜딩페이지로 설계합니다. 섹션 구성(위→아래):

1. **헤더(Sticky Nav)**: 로고, 기능/요금제 앵커 링크, 로그인, "무료로 시작하기" 버튼
2. **Hero**: 헤드라인 + 서브카피 + CTA + 실제 대시보드 스크린샷/목업
3. **문제 제기**: "이런 고민 있으신가요?" — 크리에이터·광고주·마케터 3개 pain point 카드 (1.2 페르소나 연동)
4. **핵심 기능 하이라이트**: 4~5개 기능을 스크린샷과 함께 지그재그 레이아웃으로 소개(대시보드, 트렌드 탐색, 경쟁사 광고 모니터링, AI 스튜디오)
5. **채널 지원 매트릭스 미리보기**: "인스타그램·틱톡·쓰레드, 각각 무엇을 볼 수 있나요?"를 간소화한 표 — PART 3의 축약 버전을 랜딩에도 노출해 기대치를 처음부터 정직하게 설정
6. **AI 데모 섹션**: AI 에이전트 대화 예시, 카드뉴스 생성 전/후 비교
7. **요금제 미리보기**: 3~4단 카드 요약 + "자세히 보기" → PART 9 페이지로 연결
8. **연동 채널 로고 스트립**: Instagram / TikTok / Threads / Meta 광고 아이콘 나열
9. **FAQ 아코디언**
10. **최종 CTA 섹션**
11. **Footer**: 정책 링크, 회사 정보, 소셜 채널

**반응형 기준(제안)**: Desktop(1280px+) 2컬럼 지그재그 → Tablet(768px~) 세로 스택, 네비 축약 → Mobile(~390px) 햄버거 메뉴, 단일 컬럼, 하단 고정 CTA 버튼 고려

### 6.2 로그인 후 대시보드 (사이드바 레이아웃)

요청하신 대로 로그인 후에는 사이드바 기반 레이아웃으로 설계합니다.

- **좌측 사이드바**(고정폭 약 240px, 접으면 72px 아이콘바): 로고, PART 5 메뉴 전체, 하단에 사용량 게이지 미니 위젯 + 업그레이드 버튼
- **상단바**: 채널 스위처(전체 / Instagram / TikTok / Threads 토글), 전역 검색, 알림 벨, 프로필 메뉴
- **메인 콘텐츠 영역**: 페이지별 카드 그리드 기반 콘텐츠
- **플로팅 AI 에이전트**: 우하단 고정 버튼 → 클릭 시 우측에서 챗 패널 슬라이드인(전 페이지 공통)

**반응형 기준(제안)**: Tablet은 사이드바 자동 축소(아이콘만, 클릭 시 확장) / Mobile은 사이드바 → 하단 탭바(핵심 5개 메뉴) + "더보기" 메뉴, 상단바는 채널 스위처와 알림만 남기고 간소화

## PART 7. 디자인 시스템

첨부해주신 토스증권 디자인 레퍼런스를 기본 골격으로 삼되, 두 가지는 의도적으로 다르게 가져갑니다: **① 브랜드 컬러**(메타 블루와 겹치지 않도록 색상을 이동) **② 상승/하락 색상 규칙**(주식 매수/매도 관행이 아니라 일반적인 마케팅 대시보드 관행 적용). 이유는 아래에서 각각 설명합니다.

### 7.1 왜 토스증권 스타일이 이 프로젝트에 잘 맞는가

토스증권 레퍼런스의 핵심은 "다크모드 기본 + 데이터 밀도가 높은 계기판(instrument panel) 느낌 + 장식 없는 타이포그래피"입니다. 이 프로젝트도 조회수·팔로워·광고 성과 같은 숫자 데이터를 다루는 대시보드형 SaaS이므로 구조적으로 매우 잘 맞습니다(Datadog, Vercel, Linear 같은 데이터 중심 SaaS들이 다크모드를 기본으로 채택하는 것과 같은 맥락). 다만 랜딩페이지처럼 신규 방문자에게 "따뜻하고 친근한" 인상을 줘야 하는 영역에서는 다크모드를 유지하되 브랜드 컬러의 채도·명도를 조금 더 밝게 써서 균형을 잡는 것을 권장합니다.

### 7.2 다크모드 서페이스 구조 (토스 구조 차용)

| 레이어 | 값 | 역할 |
|---|---|---|
| Surface 100 (최심층) | `#0C0C11` | 페이지 최하단 배경 |
| Body | `#16161C` | 실제 렌더링되는 본문 배경 |
| Overlay 300 | `#212128` | 모달·시트·드롭다운 배경 |
| 패널 테두리 | `rgba(255,255,255,0.08)` | 카드·패널 구분선(그림자 대신 사용) |

- **그림자(box-shadow) 미사용** — 토스와 동일하게 레이어드 알파 + 반투명 테두리로 깊이감 표현
- **라운드 2단계**: 카드·버튼·인풋 `8px` / 칩·필터·뱃지 `32px` (3단계 이상으로 늘리지 않음)

### 7.3 브랜드 컬러 — 시그널 코랄 (최종 확정)

**시그널 코랄(Signal Coral)** 로 확정합니다.

| 토큰 | 값 | 용도 |
|---|---|---|
| Primary Default | `#FF6B4A` | 주요 CTA, 브랜드 아이콘, 로고 |
| Primary Hover | `#E85A3B` | 호버 상태 |
| Primary Pressed | `#C4472C` | 클릭 상태 |
| **On-Primary(텍스트)** | **`#0C0C11`(다크 텍스트)** | 코랄 배경 위 텍스트 — 아래 참고 |

**⚠️ 확정하면서 바로잡은 부분**: 지난 제안에서 온컬러 텍스트를 "화이트 또는 다크, 대비 확인 후 결정"으로 남겨뒀는데, 실제로 계산해보니 흰색 텍스트는 이 코랄 위에서 명도 대비가 약 2.8:1로 WCAG AA 기준(일반 텍스트 4.5:1, 큰 텍스트·버튼 등 UI 컴포넌트 3:1)에 못 미칩니다. 반면 다크 텍스트(`#0C0C11`)는 약 6.9:1로 여유 있게 통과합니다. 그래서 **코랄 배경 위에는 항상 다크 텍스트를 쓰는 것으로 확정**합니다 — 버튼·배지·아이콘 등 모든 코랄 CTA에 공통 적용해 주세요.

**확정 배경**: 이 카테고리(SNS 분석·메타광고 도구)의 경쟁 서비스는 거의 전부 파랑·검정·보라 계열입니다(Socialinsider, Modash, 각종 애드테크 툴 포함). 다크모드 배경(`#0C0C11`) 위에 웜톤 코랄을 올리면 스크린샷 한 장만으로도 다른 서비스와 확실히 구분됩니다. "차가운 분석 도구"가 아니라 "AI 에이전트가 다정하게 브리핑해준다"는 핀치의 포지셔닝(4.9 AI 에이전트)과도 방향이 맞고, 무엇보다 "핀치(작은 새)"라는 이름의 따뜻하고 가벼운 느낌과도 톤이 자연스럽게 이어집니다.

> 검토했으나 채택하지 않은 대안: 일렉트릭 인디고(`#5B4FFF`, "AI 하면 떠오르는" 더 전형적인 블루퍼플 계열) — 기록 차원에서만 남겨둡니다.
>
> 최종 확정 전 상표·기존 브랜드 색상 충돌 여부를 간단히 검색해보시는 걸 권장드립니다. 로고 시안이 나오면 PART 6 페이지 목업에 실제로 대입해보고 색조를 미세 조정하시길 권합니다.

### 7.3-1 로고·아이콘 방향

"핀치(작은 새)"는 로고 확장성이 좋은 이름입니다. 새의 실루엣을 딴 심볼 마크를 시그널 코랄 단색으로 만들고, 파비콘·앱 아이콘에서는 실루엣만 축약해 쓰는 방향을 제안합니다. 새가 지저귀는 모습을 "알림"이나 "트렌드가 지금 막 도착했다"는 느낌(4.6 경쟁사 광고 모니터링, 4.12 알림센터)과 자연스럽게 연결할 수 있어, 로딩 애니메이션이나 알림 아이콘에도 같은 모티프를 재사용하면 브랜드 일관성을 쉽게 확보할 수 있습니다.

### 7.4 시맨틱 컬러 — 상승/하락 규칙 (토스 관행과 의도적으로 다르게)

첨부 레퍼런스는 한국 주식시장 관행에 따라 **상승=빨강, 하락=파랑**을 쓰고 있고, 문서 자체에도 "이 규칙은 KR/JP/TW 금융 UI에 한정되며 다른 도메인에 그대로 이식하면 안 된다"는 경고가 명시되어 있습니다. 이 프로젝트는 주식 거래 도구가 아니라 **마케팅 성과 대시보드**이므로, 한국 사용자 대상이라도 주식 관행이 아니라 구글 애널리틱스·메타광고 매니저 등 마케팅 도구들이 공통으로 쓰는 **상승=초록, 하락=빨강**을 적용하는 것을 권장합니다. (레퍼런스 문서의 "지역별 관행"이 아니라 "도메인별 관행"의 문제이기 때문입니다.)

| 토큰 | 값 | 용도 |
|---|---|---|
| Positive(성장) Default | `#22C55E` | 팔로워 증가, 조회수 상승 등 |
| Positive Hover | `#16A34A` | — |
| Positive Weak(배경) | `rgba(34,197,94,0.15)` | 상승 행 배경 틴트 |
| Negative(하락) Default | `#F04452` | 팔로워 감소, 참여율 하락 등 |
| Negative Weak(배경) | `rgba(240,68,82,0.15)` | 하락 행 배경 틴트 |
| Warning(주의) | `#F5A524` | 예산 소진 임박, 토큰 만료 임박 등 |

### 7.5 채널 배지 컬러 (브랜드 컬러와 별도 관리)

대시보드 곳곳에서 "이 데이터가 어느 채널 것인지" 한눈에 구분되어야 하므로, 브랜드 프라이머리 컬러와는 별도로 채널 전용 배지 컬러를 정의합니다.

| 채널 | 배지 컬러 | 비고 |
|---|---|---|
| Instagram | `#E1306C` | 공식 그라디언트 대신 대표색 단색 사용(작은 배지에서는 그라디언트가 지저분해 보임) |
| TikTok | 배경 `#000000` + 포인트 `#25F4EE` / `#FE2C55` | 틱톡 공식 듀오톤 유지 |
| Threads | `#000000` / `#FFFFFF` (모노톤) | 원 브랜드가 무채색이므로 그대로 존중 |
| Meta 광고 | `#0866FF` | 광고 관리 영역 한정 사용(다른 곳에서는 브랜드 프라이머리와 혼동되지 않게 사용 자제) |

> **코랄 확정 후 체크포인트**: 브랜드 프라이머리(코랄 `#FF6B4A`)와 인스타그램 배지(`#E1306C`)는 둘 다 따뜻한 색 계열이라 색상각이 약 30도 정도로 가깝습니다. 인스타그램 카드 안에 코랄 CTA 버튼이 같이 놓이는 화면(예: 4.5 타계정 분석의 인스타그램 탭)에서는 실제 목업으로 나란히 놓고 구분이 잘 되는지 한 번 확인해보시길 권합니다.

### 7.6 타이포그래피

토스증권은 자체 제작 폰트(Toss Product Sans)를 쓰지만 외부에 공개되어 있지 않으므로, 오픈소스이면서 국내 스타트업이 가장 널리 쓰는 한글 웹폰트인 **Pretendard**를 대안으로 제안합니다. 국문·영문·숫자 밸런스가 좋고 라이선스가 자유로워 SaaS 제품에 적합합니다. 숫자(조회수, 팔로워 수 등)는 반드시 tabular-nums(고정폭 숫자)를 적용해 표에서 자릿수가 흔들리지 않도록 합니다.

| 역할 | 크기/굵기 | 용도 |
|---|---|---|
| Section H2 | 24px / 700 | 섹션 제목("대시보드", "경쟁사 분석" 등) |
| Sub H3 | 19px / 700 | 서브 섹션 제목 |
| Nav | 15px / 500 | 사이드바·글로벌 내비게이션 |
| Body | 16px / 400 (모바일 15px) | 본문·버튼 텍스트 |
| Input | 15px / 400 | 폼 필드 |
| Caption/Chip | 12px / 600 | 배지, 필터 칩, 태그 |

### 7.7 Do / Don't 요약

**Do**
- 다크모드 우선(최심층 `#0C0C11` → 본문 `#16161C` → 오버레이 `#212128`), 그림자 대신 반투명 테두리로 깊이 표현
- 라운드는 카드/버튼/인풋 8px, 칩/뱃지 32px 두 단계만 유지
- 숫자 데이터에는 tabular-nums 적용
- 채널 배지 컬러(7.5)와 브랜드 프라이머리(7.3)를 명확히 분리해서 사용
- 3rd party 데이터가 섞인 화면에는 반드시 출처·최신성 표기(PART 3, 4.4 참고)

**Don't**
- 상승/하락에 주식 관행(빨강=상승/파랑=하락)을 적용하지 않기 — 이 제품 도메인에는 초록/빨강이 맞음(7.4)
- 브랜드 프라이머리로 메타 블루(`#0866FF`)에 가까운 색을 그대로 쓰지 않기 — 자체 브랜드감이 옅어짐
- 라운드 3단계 이상으로 늘리지 않기
- 데이터 카드에 장식적 일러스트 넣지 않기 — 차트·숫자 자체가 비주얼이 되도록

## PART 8. 브랜드 네이밍 — 핀치(Finch) (최종 확정)

**핀치(Finch)** 로 확정합니다. 검토했던 다른 후보와 최종 확인해야 할 사항을 아래에 정리합니다.

**컨셉**: 작은 새(되새) 모티프 — 가볍고, 빠르고, 지저귀듯 소식을 물어다 준다는 이미지가 4.4(실시간 트렌드), 4.6(경쟁사 광고 자동 모니터링), 4.12(알림센터)처럼 "새로운 소식을 놓치지 않고 물어다 주는" 이 서비스의 핵심 기능들과 자연스럽게 맞아떨어집니다. 로고 확장성도 좋아 PART 7.3-1에 아이콘 방향을 별도로 정리해뒀습니다.

**⚠️ 확정 전 실제 확인한 사항**: 검색해보니 국내에 동일한 이름(핀치)을 쓰는 SNS 분석 서비스는 확인되지 않았습니다. 다만 해외에 **FinchAI(finchai.com)** 라는, AI 기반으로 대규모 데이터에서 관계·트렌드·인사이트를 찾아주는 기업용 분석 툴이 존재합니다. 국가안보·기업 인텔리전스 분야가 주력이라 타겟 시장은 다르지만, "AI + 데이터 분석 + 인사이트"라는 컨셉이 근접해 있어 참고해 주세요. 이 정도 근접도는 실제 서비스 등록·상표 출원 전에 키프리스 상표 검색과 도메인 확인을 통해 한 번 더 짚고 넘어가시는 걸 권장드립니다(아래 체크리스트).

## PART 13. SEO · GEO · 보안 가이드라인 (전 페이지 공통 적용)

PART 7(디자인 시스템)이 "모든 페이지가 지켜야 할 시각 규칙"이라면, 이 PART는 **"모든 페이지가 지켜야 할 기술 규칙"**입니다. 개별 기능(PART 4)이 아니라 PART 6에서 설계한 모든 페이지에 공통으로 깔리는 레이어이므로, Claude Code로 페이지를 만들 때마다 매번 챙기기보다 **공통 레이아웃/미들웨어 단위로 한 번에 적용**하는 것을 권장합니다(13.5 참고).

### 13.1 페이지 유형별 적용 범위

SEO·GEO는 "검색·AI에 노출되고 싶은 페이지"에만 의미가 있고, 오히려 로그인 후 대시보드처럼 개인정보가 담긴 페이지는 검색에 노출되면 안 됩니다. 반면 보안은 예외 없이 전 페이지 적용입니다.

| 페이지 유형 | 예시 | SEO | GEO | 보안 |
|---|---|---|---|---|
| 공개 마케팅 페이지 | 랜딩, 요금제, 기능소개 | ✅ 최우선 적용 | ✅ 최우선 적용 | ✅ 기본 보안 헤더 |
| 인증 페이지 | 로그인 / 회원가입 | ⚠️ 최소 적용 | ❌ 해당없음 | ✅✅ 강화(가장 공격 표적이 되기 쉬움) |
| 로그인 후 대시보드 전체 | /dashboard, /analyze 등 PART 5의 모든 사이드바 메뉴 | ❌ noindex 처리 | ❌ 해당없음 | ✅✅ 강화(인증+인가+데이터 격리) |
| 공유형 페이지 | 팀 워크스페이스 "뷰어 링크"(4.10) | ⚠️ 기본 noindex, 사용자가 명시적으로 공개 전환한 경우만 예외 | ⚠️ 공개 전환 시에만 | ✅ 추측 불가능한 토큰 URL + 만료시간 필수 |

### 13.2 SEO (검색엔진 최적화) 체크리스트

- **메타 태그**: 페이지별 고유 title·description, canonical URL 지정
- **Open Graph / Twitter Card**: 소셜 공유 미리보기 — 특히 AI 스튜디오(4.8)에서 만든 카드뉴스를 사용자가 외부에 공유할 때도 적용 고려
- **sitemap.xml**: 공개 페이지만 포함해 배포 파이프라인에서 자동 생성
- **robots.txt**: `/dashboard`, `/analyze` 등 로그인 필요 경로는 Disallow, 공개 페이지는 Allow
- **구조화 데이터(Schema.org JSON-LD)**: Organization, SoftwareApplication, FAQPage, Product/Offer(요금제 페이지)
- **시맨틱 HTML**: 페이지당 h1 1개, 논리적 heading 계층, `nav`/`main`/`article`/`section` 태그 사용
- **이미지 최적화**: alt 텍스트 전수 적용, WebP 자동 변환·lazy loading(Next.js `next/image` 활용)
- **Core Web Vitals**: 마케팅 페이지는 SSG/ISR로 정적 생성해 LCP 최소화, 이미지·폰트 사이즈 사전 지정으로 CLS 방지
- **URL 구조**: 사람이 읽기 쉬운 슬러그(`/pricing`, `/features/competitor-monitoring`)
- **내부 링크·브레드크럼** 구조화

### 13.3 GEO (생성형 엔진 최적화 · AI 검색 최적화)

SEO가 "구글 검색결과 상위 노출"이라면, GEO는 **"ChatGPT·퍼플렉시티·제미나이·구글 AI 오버뷰 같은 AI 답변 안에 이 서비스가 언급·인용되는 것"**을 목표로 하는, 2024년 이후 등장한 비교적 새로운 영역입니다. 아직 업계 표준이 완전히 정립되진 않았지만 지금부터 준비해두면 이후 유리합니다.

- **자기완결적 문장**: 각 페이지 상단에 맥락 없이도 이해되는 명확한 정의 문장을 배치(예: "핀치는 인스타그램·틱톡·쓰레드를 한 곳에서 분석하는 SNS 마케팅 도구입니다") — AI가 그대로 인용하기 좋은 형태
- **FAQ 콘텐츠 + FAQPage 구조화 데이터**: AI 엔진이 질문-답변 형식을 특히 잘 인용합니다
- **비교표를 공개 페이지에도 노출**: PART 3(채널별 기능 매트릭스) 같은 표는 "인스타그램이랑 틱톡 중 뭐가 더 나아?" 같은 AI 질의응답에 인용되기 좋은 자료입니다. 랜딩페이지 6.1의 "채널 지원 매트릭스 미리보기" 섹션이 이 역할을 겸하도록 설계돼 있습니다
- **검증 가능한 근거 명시**: "메타 광고 라이브러리 API 기반" 같은 구체적 출처를 명시하면 AI가 신뢰 신호로 활용하기 좋습니다
- **llms.txt 배치**: 2024년 하반기부터 확산 중인 신규 규약으로, AI 크롤러에게 사이트 핵심 콘텐츠를 요약해 안내하는 "robots.txt의 AI 버전"입니다. 사이트 루트(`/llms.txt`)에 배치하는 것을 권장합니다 — 아직 완전한 표준은 아니지만 선점 가치가 있습니다
- **AI 크롤러 허용 여부 결정**: GPTBot, PerplexityBot, ClaudeBot, Google-Extended 등을 robots.txt에서 허용할지 정해야 합니다. 다만 흥미로운 지점이 있는데, 이 서비스 자체가 PART 2에서 "우리 서비스가 다른 플랫폼 데이터를 함부로 긁어오면 안 된다"는 원칙을 다루면서, 동시에 "우리 사이트는 AI 검색엔진에 긁혀서 노출되고 싶다"는 반대 입장을 취하게 됩니다. 공개 마케팅 페이지는 허용하고, 사용자 데이터가 있는 페이지는 13.1에서 이미 noindex 처리되므로 자연히 AI 크롤링에서도 제외되어 이 둘은 실제로 충돌하지 않습니다
- **외부 언급 확산**: 비교 아티클, 브런치·커리어리 같은 커뮤니티에 자연스럽게 노출되는 콘텐츠 마케팅 — AI 엔진은 여러 소스에 걸쳐 일관되게 언급되는 브랜드를 신뢰 신호로 취급합니다

### 13.4 보안 (Security) 체크리스트

이 서비스는 SNS 계정 접근 권한(게시·삭제까지 가능), 광고 예산을 직접 다루는 메타광고 계정 권한, 결제 정보, 대행사 사용 시 여러 클라이언트 데이터까지 다루므로 일반적인 SaaS보다 보안 비중이 큰 유형입니다.

| 영역 | 체크리스트 |
|---|---|
| 인증·세션 | 강력한 비밀번호 정책, 2FA(Pro/Agency 플랜은 사실상 필수 권장), JWT 짧은 만료 + 리프레시 토큰 로테이션, 자동 로그아웃 |
| OAuth 토큰 관리 | DB 저장 시 암호화(AES-256), 평문 로그 절대 금지, 기능별 최소 권한(scope)만 요청(4.2와 연결) |
| API 키·시크릿 | 시크릿 매니저 사용(코드에 하드코딩 금지), 정기 로테이션 |
| 전송 구간 | HTTPS 전면 강제 + HSTS |
| 보안 헤더 | CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy — 공통 미들웨어로 일괄 적용(13.5) |
| 입력 검증 | ORM으로 SQL Injection 방지, 사용자 생성 콘텐츠(AI 에이전트 대화·카드뉴스 등) XSS 이스케이프, CSRF 토큰 |
| Rate Limiting | 로그인 시도, API 엔드포인트, 특히 비용이 드는 AI 생성 기능(4.8)에 필수 적용 — 어뷰징이 곧 비용 손실로 직결 |
| 결제 보안 | PG사(토스페이먼츠 등)에 위임, 카드정보 직접 저장 금지(PCI-DSS 범위 최소화) |
| 팀·대행사 데이터 격리 | RBAC(역할 기반 접근제어) + 클라이언트별 데이터 논리적 분리 — 4.10 팀 워크스페이스에서 특히 중요(A 클라이언트 담당자가 B 클라이언트 데이터를 보면 안 됨) |
| 감사 로그 | 광고 캠페인 생성·수정·삭제, 계정 연동·해제 등 민감 행동 기록 |
| 개인정보보호 | 개인정보처리방침 수립, 회원 탈퇴 시 연동 토큰 즉시 revoke, 쿠키 동의 배너 |
| 인프라 | WAF·기본 DDoS 방어(Vercel/Cloudflare 등 활용 시 기본 제공되는 경우 많음), 의존성 취약점 자동 스캔(Dependabot 등) |

### 13.5 구현 방식 — "모든 페이지 적용"을 실제로 보장하는 방법

체크리스트를 페이지마다 수작업으로 챙기면 반드시 누락이 생깁니다. PART 10(기술 스택)의 Next.js 기준으로, 아래처럼 **한 곳에서 정의하면 전 페이지에 자동 적용되는 구조**를 권장합니다.

- 보안 헤더(CSP 등)는 `middleware.ts` 한 곳에 정의 → 모든 요청에 자동 적용
- SEO 메타 태그는 각 페이지의 `metadata` export를 강제하는 린트 규칙 또는 공용 레이아웃의 기본값 + 페이지별 override 패턴으로 통일
- `robots.txt`·`sitemap.xml`은 라우트 목록(PART 5 사이트맵)을 기준으로 빌드 시 자동 생성 스크립트로 관리해, 새 페이지 추가 시 수동 등록을 잊는 일을 방지
- 인증이 필요한 라우트(`/dashboard/*` 전체)는 라우트 그룹 단위로 인증 가드를 한 번만 걸어 개별 페이지에서 매번 체크하지 않도록 구성

### 13.6 로드맵 반영 (PART 11과 연결)

- **Phase 1**: 기본 메타 태그·sitemap·robots.txt, 보안 헤더, HTTPS/HSTS, OAuth 토큰 암호화, 기본 Rate Limiting
- **Phase 2**: 구조화 데이터 확장, FAQ 콘텐츠 + GEO 대응 착수, 2FA, 감사 로그, RBAC 기초
- **Phase 3**: llms.txt, 팀 워크스페이스 데이터 격리 고도화, 정기 보안 점검(펜테스트) 착수

## PART 14. 확정 전 결정이 필요한 항목

서비스명(핀치)과 브랜드 컬러(시그널 코랄)까지 확정됐으니, 이 시점에서 기획서 전체를 다시 훑어 아직 열려있는 결정 사항들을 정리했습니다. 전부 개발을 막는 항목은 아니라서, **지금 정해야 하는 것 / 진행하면서 정해도 되는 것 / 기획 범위 자체를 한번 더 확인할 것**으로 나눴습니다.

### 지금 정하는 게 좋은 것 (Phase 1 착수 전)

| 항목 | 왜 지금인가 |
|---|---|
| 도메인·상표 실사 (PART 8 체크리스트) | 이름이 막 확정됐으니 지금이 가장 빠르게 움직일 수 있는 시점입니다. 미루면 나중에 이름을 또 바꿔야 할 수도 있습니다 |

### 진행하면서 정해도 되는 것 (Phase 2~3 이전까지)

- **3rd party 데이터 공급자 최종 선택 및 월 예산** (PART 2.3) — HikerAPI로 소액 테스트 후 본계약 규모를 정하는 방식을 제안했지만, 상한 예산은 사용자님만 정할 수 있는 부분입니다
- **AI 생성 영상 API 벤더** (PART 2.4) — PiAPI/Runway/Sora 계열 중 실제 계약처는 아직 미정입니다. 카드뉴스(Claude API 활용)보다 벤더별 단가 차이가 커서 요금제 원가(PART 9)에 영향을 줍니다
- **결제 PG사** (PART 10) — 토스페이먼츠/아임포트 중 미정
- **정식 요금제 금액(원화)** — PART 9는 플랜별 "기능 구성"만 제안했고 실제 월 구독료는 정하지 않았습니다
- **라이트모드 지원 여부** — PART 7은 다크모드 전용으로 설계했습니다. 나중에 라이트모드를 추가하는 것보다 필요 여부를 미리 정해 토큰 체계를 처음부터 두 벌로 준비하는 편이 훨씬 적은 공수로 끝납니다
- **네이티브 모바일 앱(iOS/Android) 계획 여부** — 지금까지는 반응형 웹만 전제로 설계했습니다. 네이티브 앱까지 고려하면 PART 10 기술스택에 React Native 등이 추가로 필요합니다
- **개인정보처리방침·이용약관** — PART 12·13에서 "법률 자문 필요"로 남겨둔 항목으로, 정식 출시 전까지만 준비되면 됩니다

### 기획 범위 자체를 한 번 더 확인할 것

- **PART 4.10~4.13**(팀 워크스페이스, 리포트 자동화, 알림센터, 사용량 관리)은 제가 "요청 범위 밖 추가 제안"으로 넣은 기능입니다. 필요성·우선순위가 실제 계획과 맞는지 한 번 훑어봐 주세요
- **PART 11 Phase 4**(대행사 멀티클라이언트, 인플루언서 매칭 등 확장 아이디어)는 옵션으로 남겨뒀습니다. 처음부터 대행사를 핵심 타겟으로 볼지, 개인 크리에이터·광고주를 먼저 공략하고 나중에 대행사로 확장할지에 따라 PART 11 로드맵의 순서 자체가 바뀔 수 있습니다
- **초기 타겟 시장**(한국 한정 vs 글로벌 염두) — 이 기획서는 한국 시장을 기준으로 작성돼 있습니다. 글로벌을 함께 고려한다면 다국어(hreflang), 통화, PG사, 그리고 Meta Ad Library의 국가별 정책 차이(PART 2.2에서 언급한 "한국은 광고비 비공개" 같은 사례)까지 다시 짚어야 합니다
